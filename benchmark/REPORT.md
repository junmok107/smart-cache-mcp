## 시나리오 설명

- **동일 질의 반복** (`01_repeated_query`): 완전히 동일한 질의를 10회 반복 호출 — 첫 호출은 캐시가 비어있어 항상 미스이므로, 나머지 9회 반복 호출의 적중률 100%(=전체 10회 중 90%)가 목표
- **유사 질의 변형** (`02_paraphrased_query`): 같은 의미를 다른 표현으로 물어보는 질의 10개 — 퍼지 매칭(코사인 유사도 0.90 임계값) 검증
- **다국어 질의** (`03_multilingual_query`): 같은 의미를 한국어/영어로 번갈아 질의 — multilingual-e5-base의 다국어 성능 검증
- **혼합 워크로드** (`04_mixed_workload`): 신규 질의 50% + 직전 질의 반복 50%를 번갈아 호출 — 실제 사용 패턴 시뮬레이션

## 지연시간에 대한 솔직한 해석

표를 보면 **캐시 hit(13~18ms)가 캐시 미사용 기준선(1.5~3ms)보다 오히려 느립니다.** 숨기지 않고 그대로 밝힙니다: 이 벤치마크의 다운스트림 지식조회 서버는 인메모리 키워드 매칭만 하는 목업이라 사실상 즉시 응답하는 반면, 캐시 hit는 매번 임베딩 서비스에 질의 임베딩을 요청하고(`query:` 프리픽스, GPU 추론 포함) pgvector로 유사도 검색을 하는 실제 네트워크 왕복이 필요합니다 — 이 오버헤드가 목업 서버의 응답 시간보다 큽니다.

즉 이 벤치마크가 실측으로 보여주는 이득은 **토큰 절감**이며, **지연시간 이득은 원본 MCP 서버가 얼마나 무거운 작업을 하는지에 달려 있습니다.** 실제 검색 API, LLM 호출, 무거운 DB 쿼리처럼 원본 호출 자체가 수백 ms~수 초가 걸리는 경우에는 캐시 hit의 고정 오버헤드(~15ms)를 상쇄하고도 남아 지연시간도 함께 줄어들 것으로 예상되지만, 이 벤치마크의 목업처럼 원본 호출이 극단적으로 가벼운 경우에는 캐시가 지연시간 측면에서 오히려 손해일 수 있습니다.
